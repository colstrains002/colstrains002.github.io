「文章を置く場所」が欲しくなった。  
noteみたいに気軽に書けて、見た目もそれなりに整っていて、あとから少しずつ手入れできる場所。コメントもタグも別に要らない。むしろ無い方が落ち着く。トップページとプロフィールページがあって、記事が並んでくれれば十分だと思った。

ただ、ひとつだけ条件が増えた。下書きをどうするか。  
ブラウザで下書きしたいし、書きかけのものが一覧に混ざるのは困る。でも、完全に「自分しか見られない」仕組みまでは今すぐいらない。URLを知っていれば見える、くらいのゆるい運用でいい。つまり「一覧とRSSには出さない」けど「直リンクはOK」。この線でいくことにした。

本題。  
今回は GitHub Pages を使って、Astro のブログテンプレートを土台に、個人サイトを公開できるところまで持っていった。最初のゴールは単純で、テンプレのトップが公開URLでちゃんと表示されること。ここを越えないと何も始まらないので、まずはそこだけを目標にして進めた。

公開まで到達したら、次にやったのがローカル確認の整備だった。  
文章はブラウザでも書けるけれど、テンプレートの改造やレイアウト調整をするなら、手元で build が通る状態は欲しい。最初はいくつか詰まったけど、結局は「プロジェクト直下で依存を入れて、ビルドする」という当たり前の形に戻った。ここが通ると、作業の怖さが一段減る。壊した時に気づけるから。

そしてメインの下書き運用。  
draft を frontmatter に入れて、一覧とRSSからは外す。ただし記事ページ側は外さない。こうすると、一覧には出てこないのに、URLを知っていれば表示できる。いかにも「運用ルールとしての下書き」という感じで、今の自分にはちょうどいい落とし所だった。

実際に下書きを作ってみると、置き場所で一度つまずいた。  
Astroのテンプレートには「ページとしての置き場」と「ブログ記事コレクションとしての置き場」があって、なんとなく pages の方に置くと、装飾が当たらず素のMarkdownみたいになったりする。ここで「あ、いま見てるのはテンプレのブログ記事じゃないな」と気づけたのは良かった。結局、記事は content 側に統一して、テンプレの流れに乗せた。

次に、一覧に下書きが出てしまう問題が出た。  
一覧側は draft を除外するつもりで書いてあるのに、なぜか出る。こういう時は、見た目の現象から辿るより、データがどう扱われてるかを疑った方が早い。結果としては、content collection のスキーマ側に draft を定義していなかったのが原因で、draft が「存在しないもの」扱いになっていた。ここを直したら、一覧からは消えて、直リンクは見える、という状態に戻った。

ここまでで、ひとまず「公開できる」「記事が増やせる」「下書き運用ができる」まで整った。  
公開記事として、今回の流れをまとめた記事も1本アップして、実際に表示できるところまで確認した。これで「作って終わり」ではなく、「運用を始められる状態」にはなったと思う。

余談だが、今のトップページはまだテンプレの文言が残っていて、正直とんちんかんな感じがするし、サンプル記事も何本か残っている。だから次はそこを消して、自分の文章が並ぶ場所として落ち着かせる。トップは短い紹介と最新記事数本、プロフィールは必要最小限、くらいでいい気がする。まずは余計なものを掃除して、庭を作る。
